# 功能模块说明

## 项目概述

这是一个基于 LangChain 和 LangGraph 构建的多轮对话订单查询客服系统，支持文本输入、语音输入、图像上传、订单查询、RAG 知识检索和对话持久化等功能。

---

## 一、数据库模块 (Database Module)

### 1.1 订单数据库 (orders.db)

**文件位置：`demo.py` 第 56-115 行

**功能说明**：
- 存储订单信息，包括订单号、用户ID、订单状态、商品信息、物流信息和创建时间
- 提供订单查询功能
- 支持示例数据初始化

**核心表结构**：
```sql
CREATE TABLE orders (
    order_id TEXT PRIMARY KEY,      -- 订单号（主键）
    user_id TEXT,                 -- 用户ID
    status TEXT,                  -- 订单状态
    items TEXT,                   -- 商品信息
    logistics_info TEXT,            -- 物流信息
    created_at TEXT               -- 创建时间
)
```

**订单状态说明**：
- `shipped`：已发货
- `pending_payment`：待付款
- `delivered`：已送达

**核心函数**：
- `setup_database()`：初始化数据库并插入示例数据

## 二、RAG 检索模块 (RAG Module)

### 2.1 RAG 检索器初始化

**文件位置**：`demo.py` 第 121-161 行

**功能说明**：
- 实现检索增强生成（RAG）技术
- 存储客服政策知识库
- 支持基于语义相似度的文档检索
- 为 LLM 提供相关的政策知识

**知识库内容**：
1. 退款政策：自签收之日起 7 天内，且商品未拆封，可发起退款申请
2. 物流政策：满 50 美元免邮，标准配送一般为 3-5 个工作日
3. 质保政策：电子类商品享受 1 年制造商质保服务
4. 支付政策：支持信用卡、PayPal 和支付宝
5. 订单修改：订单状态变为"已发货"后不可再修改订单信息

**核心组件**：
- **Document**：LangChain 文档对象，包含 `page_content` 和 `metadata`
- **FakeEmbeddings**：演示用的嵌入模型（生产环境应替换为真实嵌入模型）
- **InMemoryVectorStore**：内存向量数据库，存储文档向量
- **Retriever**：检索器，支持 `invoke()` 方法进行检索

**核心函数**：
- `setup_rag_retriever()`：初始化 RAG 检索器

## 三、工具模块 (Tools Module)

### 3.1 订单查询工具 (check_order)

**文件位置**：`demo.py` 第 167-208 行

**功能说明**：

- 使用 `@tool` 装饰器定义为 LangChain 工具
- 参数化查询，防止 SQL 注入
- 格式化返回结果，便于 LLM 理解

**参数**：
- `order_id: str`：订单号

**返回值**：

**技术要点**：

- 格式化输出：`订单 {order_id}（{items}）：当前状态为『{status}』。物流信息：{logistics}。`

## 四、输入处理模块 (Input Processing Module)

### 4.2 图片输入处理 (process_image_input)

- 检查是否配置了 API 密钥
- 如果没有密钥，返回模拟结果用于演示
- 如果有密钥，调用真实的 OCR 服务识别图片中的文字

**参数**：
- `file_path: str`：图片文件路径，支持 .jpg, .png 等格式

**返回值**：

**技术要点**：
- 使用 Qwen-VL 多模态模型（阿里云通义千问的视觉语言模型）
- 支持从图片中识别文字（OCR）和理解图片内容
- 包含错误处理逻辑

## 五、LangGraph 状态与节点模块 (LangGraph State and Nodes Module)

### 5.1 AgentState 状态定义

**文件位置**：`demo.py` 第 331-351 行

**功能说明**：
- 定义 Agent 的状态结构
- AgentState 是一个 TypedDict，用于描述 LangGraph 中状态的数据结构
- 状态在工作流的节点之间传递，记录对话的当前状态

**字段说明**：
- `messages: Annotated[List[BaseMessage], operator.add]`：对话历史消息列表，使用 operator.add 实现消息追加
- `order_id: Optional[str]`：当前查询的订单号，可选字段（可能为空）

**技术要点**：
- 使用 TypedDict 定义类型安全的状态结构
- 使用 Annotated 为 messages 字段添加注解 operator.add
- operator.add 表示当多个节点更新 messages 时，将新消息追加到列表中

### 5.2 LLM 初始化

**文件位置**：`demo.py` 第 353-427 行

**功能说明**：
- 如果存在密钥则使用真实模型，否则使用简易规则引擎（Mock）
- 支持 OpenAI GPT-4o：真实的 LLM
- MockLLM：模拟 LLM，用于演示目的

**MockLLM 功能**：
- 根据用户输入的关键词进行简单的规则匹配
- 支持模拟工具调用（tool_calls）
- 支持模拟正常对话

**核心类**：
- `MockLLM`：模拟 LLM 类
  - `invoke()`：处理消息并返回响应
  - `bind_tools()`：绑定工具（返回 self，不做实际绑定）

### 5.3 聊天机器人节点 (chatbot_node)

**文件位置**：`demo.py` 第 435-456 行

**功能说明**：
- 这是 LangGraph 中的一个节点，负责处理用户输入并生成响应
- 根据历史消息决定要采取的动作
- 使用已经绑定工具的 LLM（llm_with_tools）
- LLM 会根据对话历史决定是否需要调用工具
- 如果需要调用工具，返回的 AIMessage 中会包含 tool_calls
- 如果不需要调用工具，直接返回 AIMessage

**参数**：
- `state: AgentState`：当前的 AgentState，包含对话历史等信息

**返回值**：
- 包含更新后的状态的字典，这里只更新了 messages 字段

### 5.4 输入处理节点 (input_processing_node)

**文件位置**：`demo.py` 第 458-507 行

**功能说明**：
- 这是 LangGraph 中的一个节点，负责在聊天机器人处理前对输入进行预处理
- 检查最后一条消息是否是用户消息
- 如果是文件路径（音频或图片），进行相应的处理
- 将处理结果转换为新的用户消息
- 返回更新后的状态

**参数**：
- `state: AgentState`：当前的 AgentState，包含对话历史等信息

**返回值**：
- 包含更新后的状态的字典，可能更新了 messages 字段

**技术要点**：
- 检查消息类型（HumanMessage）
- 根据文件扩展名判断文件类型（.wav/.mp3 是音频，.jpg/.png 是图片）
- 调用相应的处理函数（process_audio_input 或 process_image_input）
- 返回空更新（{"messages": []}）以避免 InvalidUpdateError

### 5.5 条件路由函数 (route_tools)

**文件位置**：`demo.py` 第 524-566 行

**功能说明**：
- 这是 LangGraph 中的条件路由函数，根据当前状态决定下一步执行哪个节点
- 从状态中获取最后一条 AI 消息
- 检查消息中是否包含工具调用（tool_calls）
- 如果包含工具调用，路由到 tools 节点
- 如果不包含工具调用，路由到 END 节点（结束）

**参数**：
- `state: AgentState`：当前的 AgentState，包含对话历史等信息

**返回值**：
- 字符串，表示要路由到的节点名称（"tools" 或 END）

**技术要点**：
- 支持多种状态类型（列表、字典、BaseMessage）
- 使用 hasattr 检查对象是否有 tool_calls 属性
- 检查 tool_calls 的长度是否大于 0

## 六、工作流构建模块 (Workflow Construction Module)

**文件位置**：`demo.py` 第 509-585 行

### 6.1 节点定义

- `input_proc`：输入处理节点
- `chatbot`：聊天机器人节点
- `tools`：工具执行节点（使用 LangGraph 预构建的 ToolNode）

### 6.2 边定义

- `START -> input_proc`：从 START 节点开始，先执行 input_proc
- `input_proc -> chatbot`：执行完 input_proc 后，执行 chatbot
- `chatbot -> tools/END`：条件边，根据条件路由函数的返回值决定执行哪个节点
- `tools -> chatbot`：执行完 tools 节点后，回到 chatbot 节点继续处理

### 6.3 检查点持久化

- 使用 SqliteSaver 将对话状态保存到 SQLite 数据库
- 这样可以实现对话中断后恢复和多轮对话上下文管理
- `check_same_thread=False`：允许多线程访问
- `checkpointer=memory`：使用 SqliteSaver 作为检查点持久化

## 八、关键核心代码详解

### 8.1 LangGraph 工作流构建（第 509-585 行

```python
# 定义图工作流
workflow = StateGraph(AgentState)

# 添加节点
workflow.add_node("input_proc", input_processing_node)
workflow.add_node("chatbot", chatbot_node)
workflow.add_node("tools", ToolNode(tools))

# 添加边
workflow.add_edge(START, "input_proc")
workflow.add_edge("input_proc", "chatbot")

# 条件边
workflow.add_conditional_edges("chatbot", route_tools)
workflow.add_edge("tools", "chatbot")

# 检查点持久化
conn = sqlite3.connect("checkpoints.db", check_same_thread=False)
memory = SqliteSaver(conn)

# 编译工作流
app = workflow.compile(checkpointer=memory)
```

**详解**：
- **StateGraph**：创建状态图，指定状态类型为 AgentState
- **add_node**：添加节点到工作流
- **add_edge**：添加边连接节点，定义执行顺序
- **add_conditional_edges**：添加条件边，根据条件路由函数的返回值决定执行哪个节点
- **SqliteSaver**：将对话状态保存到 SQLite 数据库
- **compile**：编译工作流为可执行的应用

### 8.2 流式执行工作流（第 632-652 行）

```python
# 以流式方式运行图
events = app.stream(
    {"messages": [HumanMessage(content=msg_content)]},
    config,
    stream_mode="values"
)

# 处理流式输出的事件
for event in events:
    if "messages" in event:
        last_msg = event["messages"][-1]
        if isinstance(last_msg, AIMessage):
            print(f"客服：{last_msg.content}")
            if hasattr(last_msg, 'tool_calls') and last_msg.tool_calls:
                 print(f"  [工具调用]：{last_msg.tool_calls[0]['name']}")
```

**详解**：
- **app.stream**：以流式方式运行工作流，返回一个生成器
- **stream_mode="values"**：获取状态更新
- **config**：配置参数，包含 thread_id，实现对话隔离
- **HumanMessage**：用户消息对象
- **AIMessage**：AI 消息对象，可能包含 tool_calls
- **tool_calls**：工具调用信息，包含工具名称和参数
